# Контекст проєкту: система обліку для спортзалу

## Загальна ідея

Є локальний спортзал, у якому частина обліку зараз ведеться вручну. Власник/адміністрація вже мали ідею зробити систему по картках: картки були замовлені, але сама система так і не була реалізована.

Мета першої версії — не робити великий мобільний додаток чи складну CRM, а створити простий внутрішній web app для адміністратора/власника залу.

Система має допомогти швидше знаходити клієнтів, бачити стан абонемента, фіксувати відвідування та оплати.

## Важливі факти

- Формат рішення: сайт / web app, не мобільний додаток.
- Картки вже є або були замовлені.
- На картках, схоже, просто написаний номер.
- QR-коду, штрихкоду або NFC може не бути.
- Це не проблема: номер картки можна використовувати як ідентифікатор клієнта.
- Оплати зараз готівкою.
- Банківські інтеграції, термінали, онлайн-оплати не потрібні в першій версії.
- Система має бути простою для адміністратора.
- Ймовірно, основний пристрій — телефон, планшет або ноутбук на рецепції.

## Основний сценарій

1. Клієнт приходить у зал.
2. Дає картку або називає номер / ім’я / телефон.
3. Адміністратор вводить номер картки або шукає клієнта.
4. Система відкриває профіль клієнта.
5. Адміністратор бачить:
   - ім’я клієнта;
   - номер картки;
   - телефон;
   - активний абонемент;
   - дату закінчення абонемента;
   - залишок занять, якщо абонемент по кількості;
   - статус оплати;
   - історію останніх відвідувань.
6. Адміністратор натискає “відмітити візит”.
7. Якщо треба, адміністратор додає оплату або продовжує абонемент.

## Перша версія

У першу версію варто включити:

- клієнти;
- номер картки;
- пошук по номеру картки, імені або телефону;
- типи абонементів;
- активний абонемент клієнта;
- відвідування;
- готівкові оплати;
- список абонементів, які скоро закінчуються;
- простий денний звіт по оплатах.

## Що НЕ входить у першу версію

На старті не робити:

- мобільний додаток;
- турнікети;
- автоматичний контроль входу;
- NFC/QR, якщо поточні картки без них;
- банківські інтеграції;
- онлайн-оплати;
- складну бухгалтерію;
- велику CRM;
- автоматичні повідомлення клієнтам, якщо це не підтверджено як важлива потреба.

## Потенційні ролі

### Адміністратор

Може:

- шукати клієнтів;
- додавати клієнтів;
- прив’язувати номер картки;
- дивитись абонемент;
- відмічати візити;
- додавати оплату;
- продовжувати абонемент.

### Власник

Може:

- бачити всіх клієнтів;
- бачити оплати;
- бачити звіти;
- бачити активні/неактивні абонементи;
- бачити клієнтів, у яких абонемент скоро закінчується.

## Основні сутності

### Client

Клієнт залу.

Поля:
- id
- name
- phone
- card_number
- notes
- created_at

### MembershipType

Тип абонемента.

Поля:
- id
- name
- duration_days
- visits_limit
- price
- is_active

### Membership

Абонемент конкретного клієнта.

Поля:
- id
- client_id
- membership_type_id
- start_date
- end_date
- visits_total
- visits_used
- status
- created_at

### Visit

Відвідування клієнта.

Поля:
- id
- client_id
- membership_id
- visited_at
- created_by

### Payment

Оплата.

Поля:
- id
- client_id
- membership_id
- amount
- method
- paid_at
- created_by
- comment

На старті method може бути тільки `cash`.

## Корисні екрани

### Головна сторінка адміністратора

- пошук клієнта;
- швидкий список “абонементи закінчуються скоро”;
- кнопка “додати клієнта”;
- список останніх візитів.

### Профіль клієнта

- дані клієнта;
- номер картки;
- активний абонемент;
- статус: активний / закінчився / нема абонемента;
- кнопка “відмітити візит”;
- кнопка “додати оплату”;
- історія візитів;
- історія оплат.

### Звіти

- оплати за день;
- оплати за місяць;
- кількість візитів за день;
- абонементи, які скоро закінчуються;
- клієнти, які давно не ходили, якщо це буде потрібно.

## Бізнес-цінність

Система має дати:

- менше ручного обліку;
- швидший пошук клієнта;
- менше плутанини з абонементами;
- контроль активності абонемента;
- історію оплат;
- розуміння, скільки готівки було прийнято за день;
- можливість використати вже замовлені картки.

## Ключовий принцип

Не будувати “велику систему для спортзалу” одразу.

Спочатку зробити маленьку робочу версію:

**клієнт → картка → абонемент → візит → оплата**

Після цього дивитись, що реально потрібно додавати.
